## Summary

Production-ready Dockerization for the Eventy monorepo. Your repo is **not** the flat `/backend` + `/frontend` you described — it's a **.NET 9 Clean Architecture layout** at the repo root (`Eventy.WebApi` + `Application` + `Domain` + `Infrastructure` siblings) with Angular 19 in `EventiyApp/`. The Dockerfiles and build contexts below are designed around that real structure.

## Files to create / modify

### 1. `Dockerfile.backend` (repo root)
- **Build context = repo root** (`docker build -f Dockerfile.backend .`). Required because `Eventy.WebApi.csproj` references `..\Application\Application.csproj`, etc.
- **Stage 1 `base`**: `mcr.microsoft.com/dotnet/aspnet:9.0` → sets `ASPNETCORE_URLS=http://+:8080`, `ASPNETCORE_HTTP_PORTS=8080`, exposes 8080. Running as non-root `app` user (`aspnet:9.0` image already ships UID 1654).
- **Stage 2 `restore`**: copies `*.sln` + all four `*.csproj` files **individually** (not `COPY . .`) so restore is layer-cached unless a csproj changes. Runs `dotnet restore Eventiy.sln`.
- **Stage 3 `build`**: copies full source, runs `dotnet build Eventiy.sln -c Release -o /app/build --no-restore`.
- **Stage 4 `publish`**: `dotnet publish Eventy.WebApi/Eventy.WebApi.csproj -c Release -o /app/publish --no-restore` (publish the WebApi project specifically, since it's the entry point and pulls the rest in).
- **Stage 5 `final`**: `FROM base`, `WORKDIR /app`, `COPY --from=publish /app/publish .`, creates `/app/wwwroot/uploads` with correct ownership so `UseStaticFiles` can write uploads at runtime. `ENTRYPOINT ["dotnet", "Eventy.WebApi.dll"]`.

### 2. `Dockerfile.frontend` (repo root, builds from `EventiyApp/`)
- **Build context = repo root** for consistency with GitLab CI, but only the `EventiyApp/` subtree is COPYed (the `.dockerignore` keeps `node_modules`/`dist` out).
- **Stage 1 `builder`**: `node:22-alpine` (Angular 19 needs Node ≥18; 22 is current LTS). Sets `WORKDIR /app`, copies `EventiyApp/package*.json`, runs `npm ci --no-audit --no-fund`. Copies `EventiyApp/` subtree, runs `npm run build -- --configuration production`.
- **Stage 2 `final`**: `nginx:alpine`. Removes default conf, copies `nginx.conf` → `/etc/nginx/conf.d/default.conf`, copies `EventiyApp/dist/eventiy-app/browser/` → `/usr/share/nginx/html` (the Angular 19 application builder emits under a `browser/` subdir). Runs as non-root via `nginxinc/nginx-unprivileged` alternative noted in README. `EXPOSE 8080`.

### 3. `EventiyApp/nginx.conf`
- Listens on 8080 (matches `nginx-unprivileged` / non-root).
- `root /usr/share/nginx/html; index index.html;`
- SPA fallback: `try_files $uri $uri/ /index.html;` (fixes refresh 404).
- Static assets cached (Angular hashes them, so we cache aggressively + immutable).
- **Reverse proxy** `location /api/` → `proxy_pass http://backend:8080;` with `proxy_set_header Host/X-Real-IP/X-Forwarded-For/X-Forwarded-Proto`, WebSocket upgrade headers (your `Program.cs` uses `UseWebSockets` + `WebSocketMiddleware`), and `proxy_read_timeout`. Uses `$EVENTY_API_UPSTREAM` env var with `backend:8080` default so you can override at deploy time without rebuilding.

### 5. `.dockerignore` (repo root)
Excludes `**/bin/`, `**/obj/`, `**/node_modules/`, `EventiyApp/dist/`, `.git/`, `.vs/`, `.vscode/`, `**/*.user`, `**/TestResults/`, `EventiyApp/.angular/`. This shrinks both build contexts dramatically and speeds up CI.

### 6. `.gitlab-ci.yml` (repo root)
- **Stages**: `build-test-backend`, `build-test-frontend`, `docker-build-push`.
- Uses `image: docker:24` with `services: docker:24-dind` for Docker-in-Docker.
- Backend job: uses `mcr.microsoft.com/dotnet/sdk:9.0` to `dotnet test Eventiy.sln` (tests live in `tests/`), then the docker stage builds & pushes.
- Docker stage: `docker buildx build -f Dockerfile.backend -t $AZURE_ACR_SERVER/eventy-backend:$CI_COMMIT_SHORT_SHA -t $AZURE_ACR_SERVER/eventy-backend:latest .` and the same for frontend. Uses `--cache-from` for layer caching across runs.
- Logs in via `docker login -u $AZURE_ACR_USERNAME -p $AZURE_ACR_PASSWORD $AZURE_ACR_SERVER`.
- Rules: only on `master` and tags.
- Documents the required CI/CD variables (`AZURE_ACR_SERVER`, `AZURE_ACR_USERNAME`, `AZURE_ACR_PASSWORD`).

## Build & deploy contract (documented in README additions)
- Backend image listens on **8080** (non-root), exposes 8080. Behind TLS terminator at Azure.
- Frontend image listens on **8080**, proxies `/api/*` to `http://backend:8080` by default (override via `$EVENTY_API_UPSTREAM`).
- Both use multi-stage builds → final images contain only runtime + artifacts (~100–150 MB each).

## Trade-offs / notes
- Choosing **root build context** for both Dockerfiles (rather than per-subfolder) keeps the GitLab CI `docker build` invocation uniform and is *required* for the backend (cross-project references). The `.dockerignore` ensures the frontend build doesn't ship `node_modules`.
- **Nginx reverse-proxy** means the frontend image is the public entry point. The backend container must NOT be exposed directly to the internet (only to the frontend via the container network). This matches your `environment.prod.ts` (`apiUrl: '/api'`, relative).
- I'll only **edit** `Program.cs` for the HTTPS fix; everything else is new files, so risk of breaking existing local dev is minimal. The HTTPS change is gated so local `dotnet run` still redirects to HTTPS as before.
